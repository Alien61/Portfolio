Projet 07 — Sécurité

Configuration SSH par clé publique

Générer une paire de clés SSH sur Windows et déposer la clé publique sur une VM Debian pour se connecter sans mot de passe, de façon sécurisée.

ClientWindows 10/11
ServeurVM Debian
Durée~20 minutes
Algo cléed25519 / RSA 4096
Sommaire

  7. Test de connexion

Principe de l'authentification par clé

Plutôt que de s'authentifier par mot de passe, on utilise une paire de clés cryptographiques :

  • Clé privée — reste sur le poste client, jamais partagée
  • Clé publique — déposée sur le serveur dans authorized_keys

Avantages :

  • Plus sécurisé qu'un mot de passe (pas de brute-force possible)
  • Connexion sans saisir de mot de passe à chaque fois
  • Indispensable pour l'automatisation (scripts, CI/CD, n8n…)

Prérequis

  • Windows 10/11 avec OpenSSH Client activé (installé par défaut depuis 2018)
  • Une VM Debian avec openssh-server installé et actif
  • La connectivité réseau entre le poste et la VM (ping fonctionne)
  • Un compte utilisateur sur la VM Debian

Vérifier OpenSSH côté Windows :

Get-WindowsCapability -Online | Where-Object Name -like 'OpenSSH*'

Installer OpenSSH Server sur la VM Debian si besoin :

sudo apt update
sudo apt install openssh-server -y
sudo systemctl enable --now ssh
sudo systemctl status ssh

Génération de la paire de clés (sur Windows)

Ouvrir PowerShell et lancer :

ssh-keygen -t ed25519 -C "alienis@portfolio"

Ou pour une clé RSA 4096 bits (compatibilité maximale) : 

ssh-keygen -t rsa -b 4096 -C "alienis@portfolio"

À chaque question :

  • Enter file in which to save the key : appuyer sur Entrée (emplacement par défaut C:\Users\<user>\.ssh\id_ed25519)
  • Enter passphrase : laisser vide OU saisir une phrase de passe (plus sûr mais il faudra la taper à chaque fois)

Deux fichiers sont créés :

FichierRôle
id_ed25519Clé privée — à protéger comme un mot de passe
id_ed25519.pubClé publique — à déposer sur le serveur

Récupérer la clé publique

Afficher le contenu de la clé publique pour la copier :

Get-Content $env:USERPROFILE\.ssh\id_ed25519.pub

Ou simplement :

cat ~/.ssh/id_ed25519.pub

La sortie ressemble à :

ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... alienis@portfolio

Sélectionner et copier toute cette ligne, d'un seul tenant.

Déposer la clé publique sur la VM Debian

Se connecter en SSH à la VM avec le mot de passe (dernière fois !) :

ssh alienis@10.0.0.20

Une fois connecté, créer le dossier .ssh si nécessaire :

# Créer le dossier .ssh dans le home de l'utilisateur
mkdir -p ~/.ssh

# Éditer (ou créer) le fichier authorized_keys
nano ~/.ssh/authorized_keys

Coller la clé publique copiée depuis Windows, sur une seule ligne. Sauver : Ctrl+O → Entrée → Ctrl+X.

💡

Le fichier authorized_keys peut contenir plusieurs clés publiques (une par ligne) — pratique pour autoriser plusieurs postes.

Droits sur le dossier .ssh

SSH est très strict sur les permissions. Si le dossier ou le fichier sont trop ouverts, la connexion par clé est refusée sans explication claire. 

chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chown -R $USER:$USER ~/.ssh
ÉlémentDroitsSignification
~/.ssh/700Seul le propriétaire accède
authorized_keys600Lecture/écriture propriétaire uniquement
id_ed25519 (privée, côté Windows)600Idem — à protéger

Test de connexion

Depuis PowerShell Windows :

ssh alienis@10.0.0.20

Si tout est bien configuré, la connexion s'ouvre sans demander de mot de passe.

En cas d'échec — diagnostic

ssh -v alienis@10.0.0.20

Le mode verbose (-v) indique à quelle étape ça échoue :

  • Offering public key → la clé est bien proposée
  • Server accepts key → la clé est acceptée
  • Si échec : vérifier les droits sur .ssh côté VM

Durcissement — désactiver le mot de passe

Une fois la clé fonctionnelle, on peut désactiver complètement l'authentification par mot de passe pour forcer l'usage des clés. 

sudo nano /etc/ssh/sshd_config

Modifier (ou ajouter) ces lignes :

PasswordAuthentication no
ChallengeResponseAuthentication no
PubkeyAuthentication yes
PermitRootLogin prohibit-password

Redémarrer le service SSH :

sudo systemctl restart ssh
⚠️

Ne pas fermer la session SSH en cours avant d'avoir testé la reconnexion par clé dans un autre terminal. Sinon, en cas de problème, accès perdu.

✅

Le serveur n'accepte désormais que les connexions par clé publique. Les attaques par force brute sur les mots de passe deviennent impossibles.
